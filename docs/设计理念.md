# 设计理念 / Design Philosophy

## 为什么要做这个插件 / Why This Plugin

### 背景 / Background

HarmonyOS（鸿蒙）开发者目前主要依赖华为官方的 **DevEco Studio**（基于 IntelliJ）进行开发。然而：

- DevEco Studio 体积庞大（约 2-3GB），启动缓慢
- 前端开发者更习惯 VS Code 的轻量编辑体验
- VS Code 生态中缺乏对 ArkTS/ArkUI 的原生支持
- 团队可能同时维护 Web + HarmonyOS 项目，频繁切换 IDE 降低效率

HarmonyOS developers currently rely on Huawei's **DevEco Studio** (IntelliJ-based). However, it's heavy (~2-3GB), slow to start, and frontend developers prefer VS Code's lightweight experience. There was no native ArkTS/ArkUI support in the VS Code ecosystem.

### 设计目标 / Design Goals

| 目标 / Goal | 具体指标 / Metric |
|-------------|-----------------|
| **极致轻量** / Ultra-lightweight | 生产包 < 100KB，激活时间 < 50ms |
| **按需加载** / Load on demand | 未使用功能零开销 |
| **全链路覆盖** / Full coverage | 编码 → 构建 → 运行 → 调试 → 检查 |
| **可扩展** / Extensible | 第三方插件可通过 Public API 集成 |
| **零配置** / Zero config | 打开项目即可使用，自动检测环境 |

## 核心设计决策 / Core Design Decisions

### 1. 微内核架构（而非单体架构）/ Microkernel (Not Monolithic)

**为什么 / Why**：VS Code 插件的激活速度直接影响用户体验。如果所有功能在 `activate()` 中同步加载，将导致启动延迟。

**方案 / Solution**：采用微内核 + 4 层懒加载：

```
┌─────────────────────────────────────────────────────────┐
│                    extension.ts                          │
│                  (入口协调器 / Entry Orchestrator)        │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  Layer 0 — 立即加载 / Immediate (~1ms)                   │
│  ├── EventBus     (事件通信)                             │
│  ├── Registry     (服务注册)                             │
│  └── Logger       (日志系统)                             │
│                                                          │
│  Layer 1 — 按需触发 / On first .ets file open            │
│  ├── CompletionProvider   (自动补全)                     │
│  ├── HoverProvider        (悬浮文档)                     │
│  ├── CodeLensProvider     (代码透镜)                     │
│  ├── ColorProvider        (颜色预览)                     │
│  ├── ResourceCompletion   ($r() 补全)                    │
│  ├── ResourceDefinition   ($r() 跳转)                    │
│  └── ResourceDiagnostic   ($r() 校验)                    │
│                                                          │
│  Layer 1.5 — 延迟导入 / Deferred import                  │
│  ├── DeviceTreeView     (设备面板)                        │
│  ├── TaskProvider       (构建任务)                        │
│  └── DebugProvider      (调试适配器)                      │
│                                                          │
│  Layer 3 — 命令触发 / On command invocation               │
│  ├── Build & Run        (构建运行)                       │
│  ├── UI Inspector       (UI 检查器)                      │
│  ├── Project Wizard     (项目向导)                        │
│  ├── Code Actions       (代码操作)                       │
│  └── ... (共 21 个命令)                                  │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**关键实现 / Key Implementation**：

```typescript
// Layer 1: 使用 VS Code 原生懒加载 — 注册 provider 是同步的，
// 但 provideXxx 回调内部用 dynamic import 实现真正的延迟加载
context.subscriptions.push(
  vscode.languages.registerCompletionItemProvider(LANGUAGE_ID, {
    async provideCompletionItems(document, position, token, ctx) {
      // 只有用户首次在 .ets 文件中触发补全时，才会加载模块
      const { provideCompletionItems } = await import('./language/completionProvider');
      return provideCompletionItems(document, position, token, ctx);
    },
  }, '.', '@', '\'', '"')
);

// Layer 3: 命令级懒加载
lazyCommand(COMMANDS.BUILD_AND_RUN, async () => {
  // 只有用户执行 "Build & Run" 命令时，才会加载构建模块
  const { buildAndRun } = await import('./build/buildAndRun');
  await buildAndRun({ openInspector: true });
});
```

### 2. EventBus 通信（而非直接依赖）/ EventBus (Not Direct Coupling)

**为什么 / Why**：模块间如果直接 import 互相调用，会形成耦合网络，难以维护和测试。

**方案 / Solution**：所有模块通过 EventBus 通信，完全解耦。

```typescript
// 类型安全的事件定义
export interface HarmonyEvents {
  'project:detected':    { rootPath: string; modules: string[] };
  'build:started':       { task: string; module?: string };
  'build:completed':     { task: string; success: boolean; duration: number };
  'device:connected':    { id: string; name: string; type: string };
  'device:disconnected': { id: string };
  'resource:changed':    { type: string; path: string };
  // ... 共 12 种事件
}

// 发送事件
eventBus.emit('build:completed', { task: 'assembleHap', success: true, duration: 15000 });

// 监听事件
eventBus.on('device:connected', (data) => {
  logger.info(`Device ${data.name} connected`);
});

// 模式匹配监听
eventBus.onPattern('build:*', (event, data) => {
  console.log(`Build event: ${event}`, data);
});
```

### 3. 扩展点注册表（而非硬编码）/ Extension Point Registry (Not Hardcoded)

**为什么 / Why**：为了让第三方插件能够扩展功能（添加自定义模板、构建任务、Lint 规则等）。

**方案 / Solution**：定义 8 个扩展点，通过 Registry 注册：

```typescript
export const ExtensionPoints = {
  SNIPPET:     'harmony.snippets',      // 代码片段
  SCHEMA:      'harmony.schemas',       // JSON Schema
  TEMPLATE:    'harmony.templates',     // 项目模板
  LINT_RULE:   'harmony.lintRules',     // Lint 规则
  DEVICE:      'harmony.devices',       // 设备适配器
  BUILD_TASK:  'harmony.buildTasks',    // 构建任务
  PREVIEW:     'harmony.preview',       // 预览渲染器
  CODE_ACTION: 'harmony.codeActions',   // 代码操作
};
```

**第三方使用示例 / Third-party Usage**：

```typescript
// 其他 VS Code 插件可以这样扩展
const harmonyApi = vscode.extensions.getExtension('corn12138.harmony-dev-tools')?.exports;

harmonyApi.registerTemplateContributor({
  id: 'my-iot-template',
  name: 'IoT Device App',
  description: 'Template for HarmonyOS IoT applications',
  category: 'app',
  async generate(targetDir, options) {
    // 生成 IoT 项目模板文件...
  }
});
```

### 4. HDC 而非 ADB / HDC (Not ADB)

**为什么 / Why**：HarmonyOS 使用自己的设备连接工具 HDC（HarmonyOS Device Connector），API 与 Android 的 ADB 类似但不兼容。

**关键 HDC 命令 / Key HDC Commands**：

```bash
hdc list targets          # 列出已连接设备
hdc install -r app.hap    # 安装 HAP 包（-r 覆盖安装）
hdc shell aa start -a EntryAbility -b com.example.app  # 启动应用
hdc shell aa start -D -a EntryAbility -b com.example.app  # 调试模式启动
hdc shell aa force-stop com.example.app  # 强制停止
hdc fport tcp:9230 tcp:9230  # 端口转发（调试用）
hdc shell snapshot_display -f /tmp/screenshot.png  # 截图
hdc file recv /tmp/screenshot.png ./screenshot.png  # 拉取文件
hdc hilog | grep BundleName  # 查看日志
```

### 5. hvigorw 而非 DevEco 构建系统 / hvigorw (Not DevEco Build System)

**为什么 / Why**：`hvigorw` 是鸿蒙项目的本地构建脚本（类似 Android 的 `gradlew`），可以独立于 DevEco Studio 运行。

**关键发现 / Key Findings**：

```
hvigorw 不是全局命令      → 必须使用 ./hvigorw（相对路径）
hvigorw 是 shell 脚本     → 不能用 node ./hvigorw 执行
hvigorw 可能没有执行权限  → 需要先 chmod +x ./hvigorw
```

### 6. esbuild 打包（而非 webpack）/ esbuild (Not webpack)

**为什么 / Why**：

| 指标 / Metric | esbuild | webpack |
|---------------|---------|---------|
| 打包速度 / Build speed | ~50ms | ~5000ms |
| 配置复杂度 / Config complexity | 低 / Low | 高 / High |
| 输出体积 / Output size | ~73KB | ~150KB+ |

VS Code 插件打包追求速度和体积最小化，esbuild 完胜。

## 设计约束 / Design Constraints

1. **不依赖 DevEco Studio** — 用户可以只安装 VS Code + HDC 就能开发
2. **不内置 SDK** — SDK 路径通过配置项指定或自动检测
3. **不修改用户项目** — 插件不会修改 `build-profile.json5` 等配置文件
4. **不联网** — 所有功能离线可用（搜索文档除外）
5. **最小权限** — 只请求必要的 VS Code API 权限

## 设计取舍 / Trade-offs

| 取舍 / Trade-off | 选择 / Choice | 原因 / Reason |
|------------------|--------------|--------------|
| 功能完整性 vs 轻量 | 轻量优先 | 目标是辅助开发，不是替代 DevEco |
| 类型安全 vs 开发速度 | 类型安全 | TypeScript 强类型，减少运行时错误 |
| 内联 vs 外部调试器 | 复合方案 | 自定义 DAP + 委托给 pwa-chrome |
| 全量加载 vs 懒加载 | 懒加载 | 首次激活 < 50ms |
| 自建 UI vs WebView | WebView | UI Inspector 需要复杂布局 |
