# 鸿蒙开发社区痛点 × 本插件规划

基于社区反馈（DevEco 吐槽、ArkTS 开发体验、VS Code 替代需求）整理痛点，并对照本 VS Code 插件做规划，持续迭代、不卡死。

---

## 一、社区痛点汇总

### 1. 编译与反馈慢（高频痛点）

| 痛点 | 来源 | 说明 |
|------|------|------|
| 全量编译耗时长 | 多篇社区文章 | 动辄 10–20+ 分钟，改一行等很久 |
| 热重载/增量有约束 | 官方说明 | 仅真机、单文件、单工程，不支持新增/删文件、资源 |
| 报错不直观 | 社区 | 如 "Undefined is not object" 难定位 |

**本插件已做 / 可加强：**

- ✅ **实时诊断**：不依赖 DevEco 编译，即时报 any/状态/性能问题（README 已写「无需等 20+ 分钟」）
- 🔲 在输出/问题面板中**把编译错误和实时诊断分开展示**，避免混淆
- 🔲 **错误说明增强**：对常见运行时类错误给「可能原因 + 文档链接」的说明

---

### 2. 开发效率与重复劳动

| 痛点 | 来源 | 说明 |
|------|------|------|
| 列表/页面搭骨架重复代码多 | 社区 | 几十行布局+状态+事件，耗时长 |
| 文档信息密度大、查 API 效率低 | 社区 | 新手难上手，查文档慢 |
| 相对布局 RelativeContainer 难写 | 掘金等 | 锚点手敲、无补全、无 ID 校验 |

**本插件已做 / 可加强：**

- ✅ **70+ 代码片段**：comp、entry、compv2、list、navstack 等
- ✅ **装饰器/组件补全 + 悬浮文档**：30+ 装饰器、85+ 组件
- ✅ **配置键悬浮文档**：build-profile、module、app、oh-package
- ✅ **搜索官方文档**：HarmonyOS: Search HarmonyOS Docs
- 🔲 **更多页面级片段**：列表页、Tab 页、详情页、空状态等「一整页」骨架
- 🔲 **RelativeContainer 辅助**：ID 补全、常用对齐规则片段、简单校验（可选）

---

### 3. 调试与排错

| 痛点 | 来源 | 说明 |
|------|------|------|
| 调试耗时长、断点/调用栈不顺手 | 社区 | 排查成本高 |
| 报错信息不明确 | 社区 | 难以定位根因 |

**本插件已做 / 可加强：**

- ✅ **HDC 调试**：launch 配置、端口转发、CDP 附加
- ✅ **UI Inspector**：设备截图、组件树、属性、跳转源码
- ✅ **设备日志**：View Device Logs（hdc hilog）
- 🔲 **常见错误码/信息映射**：在问题面板或 hover 里给「错误码 → 说明 + 文档」
- 🔲 **日志增强**：过滤、高亮、一键搜官方文档（可选）

### 4. 环境与配置

| 痛点 | 来源 | 说明 |
|------|------|------|
| SDK/NDK/模拟器配置繁琐 | 社区 | 路径、版本多，易出错 |
| 机械硬盘等导致模拟器/相册异常 | 踩坑记 | 环境差异大 |
| 版本兼容性（API/Beta）差异大 | 社区 | 同功能不同写法 |

**本插件已做 / 可加强：**

- ✅ **SDK/HDC 检测与配置项**：harmony.sdkPath、harmony.hdcPath
- ✅ **API 兼容性检查**：Check API Compatibility，支持 `targetSdkVersion` / `compileSdkVersion` 与用法校验
- ✅ **废弃 API 检测**：15+ 模块/函数 + 迁移建议
- ✅ **hvigor 配置检查**：modelVersion 等
- 🔲 **环境检查命令**：一键检测 SDK/HDK/模拟器/hvigor 并输出报告（可带文档链接）
- ✅ **多版本 API 文档入口（基础版）**：Docs Quick Pick 已补充官方版本说明、知识地图、Command Line Tools 入口

### 5. 学习成本与心智负担

| 痛点 | 来源 | 说明 |
|------|------|------|
| 声明式 UI + 状态管理概念复杂 | 社区 | V1/V2 两套，易混 |
| 父子组件通信限制（不能传普通函数） | 社区 | 与 Web/Android 习惯不同 |
| 数据传值/传址易错，影响 UI 更新 | 社区 | MVVM 适应成本高 |

**本插件已做 / 可加强：**

- ✅ **V1/V2 混用检测**：诊断 + 一键迁移 V1→V2
- ✅ **状态陷阱诊断**：@State 复杂对象、@Link 在 V2 等
- ✅ **Quick Fix**：any→具体类型、ForEach→LazyForEach、迁移建议
- ✅ **悬浮文档**：装饰器、配置键中英文说明
- 🔲 **「为什么这样写」简短说明**：在 hover 或 CodeLens 里加 1 句「传值/传址」「V1/V2 区别」（可逐步加）
- 🔲 **学习路径/速查**：命令或面板打开「ArkTS 速查」「状态管理对照表」等（链接到官方或自建 doc）

### 7. 工程与文件管理

| 痛点 | 来源 | 说明 |
|------|------|------|
| 多模块、多配置，心里没数 | 推断 | 哪些是入口、哪些被引用难一眼看出 |

- 🔲 **入口/路由聚合**：从 module.json5 + @Entry 聚合「可启动页面」列表（可选）

## 二、优先级与迭代原则

- **P0（立刻能减痛）**：已有能力文档化、命令好找；补 1～2 个「一键 X」。
- **P1（短期）**：更多片段、错误说明增强、环境检查命令、工程视图（用现有 fileTracker）。
- **P2（中期）**：RelativeContainer 辅助、错误码映射、学习速查入口。
- **P3（长期）**：API 跳转文档、离线文档、更复杂工程分析。

**不卡死原则：**

- 每个迭代只做 1～2 个可交付小目标。
- 先文档/配置/命令，再动大逻辑。
- 新功能优先用现有 fileTracker / 诊断 / 公共 API，避免重复造轮子。

## 四、与本项目的对照关系

| 项目能力 | 对应痛点 | 规划中的增强 |
|----------|----------|----------------|
| 实时诊断 + Quick Fix | 编译慢、报错不直观 | 错误说明增强、与编译错误区分 |
| 片段 + 补全 + hover | 重复代码多、查 API 慢 | 更多页面级片段、RelativeContainer |
| 调试 + UI Inspector + 日志 | 调试耗时、定位难 | 错误码映射、日志过滤 |
| SDK/配置 + API 兼容检查 | 环境繁琐、版本差异 | 环境检查命令、多版本文档入口 |
| V1/V2 迁移 + 状态诊断 | 学习成本、心智负担 | hover 加「为什么」、速查入口 |
| Search Docs + 配置 hover | 文档查询效率低 | 符号跳转文档、离线文档 |
| fileTracker + getProjectFileIndex | 工程/文件心里没数 | 侧栏工程视图、入口聚合 |

此文档随社区反馈和版本发布持续更新，避免规划僵化、开发卡死。
